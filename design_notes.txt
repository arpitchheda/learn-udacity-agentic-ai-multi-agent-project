============================================================
  MUNDER DIFFLIN PAPER COMPANY — MULTI-AGENT SYSTEM
  Design Notes: How the System Works
============================================================

------------------------------------------------------------
OVERVIEW
------------------------------------------------------------

This system automates three core business operations for the
Munder Difflin Paper Company using four AI agents built with
pydantic-ai 2.5.0:

  1. Inventory management  — check stock levels
  2. Quote generation      — price orders using historical data
  3. Order fulfillment     — record transactions, update cash balance

All agents share a single SQLite database (munder_difflin.db)
accessed through a common set of helper functions in db_helpers.py.


------------------------------------------------------------
AGENT ARCHITECTURE
------------------------------------------------------------

  Customer Request
        |
        v
  Orchestrator Agent  (gpt-4o, structured output)
    |        |              |
    v        v              v
Inventory  Quoting        Sales
Agent      Agent          Agent
    |        |              |
    +--------+--------------+
                 |
                 v
          SQLite Database


ORCHESTRATOR AGENT
  The single entry point for every customer request. Uses
  pydantic-ai's structured output (output_type=OrderResponse)
  to return a typed Pydantic model instead of freeform prose.
  Never queries the database directly.

  Model: gpt-4o (via openai.vocareum.com proxy)
  Output type: OrderResponse (confirmed_sales, unfulfilled_items,
               discount_rate, delivery_date)

  Responsibilities:
    - Parse the request text and extract the request date
    - Map informal customer item names to exact database names
      (e.g. "A4 glossy paper" -> "Glossy paper",
             "heavy cardstock" -> "100 lb cover stock")
    - Verify stock availability for each requested item
    - Apply tiered bulk discounts:
        0-500 units    = 0%  discount
        501-1000 units = 10% discount
        1001+ units    = 15% discount
    - Call worker agents via three tools:
        check_inventory -> inventory_agent
        record_sale     -> sales_agent
        get_delivery    -> quoting_agent
    - Return a typed OrderResponse — the customer message is
      built deterministically in Python from typed fields

  Post-processing (_fix_order_response):
    After the LLM returns OrderResponse, Python re-checks the
    discount tier from total_units. If the LLM applied the
    wrong tier, the DB transactions are deleted (by rowid) and
    re-inserted at the correct price before the customer message
    is assembled. This ensures the DB, cash balance, and
    customer response always agree.

INVENTORY AGENT
  Handles all stock-related queries.
  Model: gpt-4o-mini

    check_all_inventory   uses get_all_inventory()
      Returns full stock snapshot with unit prices as of a date.

    check_stock_level     uses get_stock_level()
      Returns units on hand for a single specific item.

QUOTING AGENT
  Estimates delivery timelines and looks up historical pricing.
  Model: gpt-4o-mini

    search_quote_history_tool   uses search_quote_history()
      Finds comparable past quotes to guide pricing.
      Filters out invalid rows where total_amount = -1.

    estimate_delivery           uses get_supplier_delivery_date()
      Calculates delivery lead time based on order quantity.
      (<=10 units: same day | <=100: +1 day | <=1000: +4 days
       | >1000: +7 days)

    get_item_unit_prices        uses inventory table directly
      Looks up exact unit prices for specific items.

SALES AGENT
  Records confirmed transactions and enforces financial controls.
  Model: gpt-4o-mini

    finalize_sale               uses create_transaction()
      Records a completed customer sale in the database.

    approve_purchase            uses get_cash_balance()
      Enforces a 20% cash safety margin before approving any
      stock purchase spend, preventing insolvency.

    check_cash_balance          uses get_cash_balance()
      Returns the current cash position as of a given date.

    calculate_financial_health  uses generate_financial_report()
      Rates business health as EXCELLENT / GOOD / FAIR / POOR
      based on the cash-to-total-assets ratio.

    generate_report             uses generate_financial_report()
      Produces a full financial summary: cash, inventory value,
      total assets, and top-selling products.


------------------------------------------------------------
ORCHESTRATION FLOW (per request)
------------------------------------------------------------

Step 1  Orchestrator calls check_inventory -> inventory_agent
        to get the full item list with stock quantities and
        unit prices as of the request date.

Step 2  Orchestrator maps informal item names in the customer
        request to exact database item names.

Step 3  Orchestrator checks that available stock >= requested
        quantity for each item. Insufficient stock -> added to
        unfulfilled_items with "only N units available
        (requested X)". Items not in catalog -> "not available
        in our catalog".

Step 4  Orchestrator calls record_sale -> sales_agent for each
        fulfillable item. record_sale looks up exact unit_price
        from inventory DB and computes:
          price = unit_price x quantity x (1 - discount_rate)
        entirely in Python before delegating to sales_agent.

Step 5  Orchestrator calls get_delivery -> quoting_agent to
        get the estimated delivery date as a YYYY-MM-DD string.

Step 6  LLM returns typed OrderResponse. _fix_order_response()
        verifies the discount tier and corrects any DB rows if
        needed. to_customer_message() then builds the response
        from typed fields — arithmetic is always correct.


------------------------------------------------------------
DESIGN DECISIONS
------------------------------------------------------------

Structured output for the orchestrator
  pydantic-ai's output_type=OrderResponse forces the LLM to
  return a validated Pydantic model rather than prose. The
  order total, discount explanation, and sign-off are all
  computed in Python @property methods — the LLM cannot
  hallucinate arithmetic.

Deterministic pricing in record_sale
  The record_sale tool does not accept a price parameter from
  the LLM. It looks up exact unit_price from the inventory
  table in Python and computes:
    price = unit_price x qty x (1 - discount_rate)
  This guarantees the DB always records the same values shown
  to the customer.

Post-processing discount correction
  _fix_order_response() re-checks the discount tier using a
  deterministic _compute_discount_rate() function after the LLM
  returns. If wrong, it deletes DB rows by SQLite rowid and
  re-inserts at the correct price. Confirmed across 5 requests
  in the test run (requests 4, 5, 7, 10, 19).

Worker agents use @agent.tool_plain
  Each worker has a small, well-defined tool set (2-5 tools).
  Tools are defined with pydantic-ai's @agent.tool_plain
  decorator and return plain strings. This is simpler, more
  predictable, and uses fewer tokens than structured output
  at the worker level.

Non-overlapping responsibilities
  The inventory agent never prices or books. The quoting agent
  never checks stock or writes transactions. The sales agent
  never reads inventory. This makes each agent independently
  testable and prevents duplicated logic.

Modular package structure
  All agent code lives in agents/ with one file per agent.
  Shared infrastructure (db_engine and all 7 helper functions)
  lives in db_helpers.py, avoiding circular imports.

20% cash safety margin
  The approve_purchase tool in the sales agent rejects any
  stock spend that would leave cash below 20% of the current
  balance. This is an automated financial control that runs on
  every proposed stock purchase without human intervention.

Retry with exponential backoff
  Transient errors (rate limit, timeout, 502/503) trigger up
  to 3 retry attempts with delays of 2 and 4 seconds. Budget
  and auth errors are non-retryable and raise immediately so
  the run continues with the next request.

Placeholder sanitisation
  A regex (_PLACEHOLDER_RE) in to_customer_message() replaces
  any LLM-generated "[Manufacturer's Name]" or "[Company Name]"
  patterns with "Munder Difflin Paper Company" before the
  response is returned to the customer.

HOW TO RUN
------------------------------------------------------------

1. Add your API key to .env:
     UDACITY_OPENAI_API_KEY=your_key_here

2. Activate the virtual environment (Windows):
     .\venv\Scripts\activate

3. Run the full evaluation:
     python project_starter.py

4. Results are written to test_results.csv

To view the workflow diagram:
  Check workflow_diagram.png or open workflow_diagram.html
  in any browser. Mermaid source is in workflow_diagram.mmd.
